O QUE FOI FEITO — TI6 / The Green Grid
=======================================

PROBLEMA ORIGINAL
-----------------
O notebook green_grid.ipynb rodava tudo em uma única passagem, com os 3 modelos
"paralelizados" via ThreadPoolExecutor. Resultado:

  - 12+ horas sem nenhuma forma de retomar se interrompido
  - Queda de energia no epoch 7 = perda total do progresso
  - ThreadPoolExecutor em CPU não ajuda de verdade: o GIL do Python bloqueia
    threads para código CPU-bound. O ganho era mínimo (~10-20% no melhor caso)
  - Uma única célula de treino para os 3 modelos tornava impossível saber qual
    travou ou retomar só um deles

O QUE FOI REORGANIZADO
-----------------------

1. CHECKPOINT POR ÉPOCA (mudança principal)
   Cada treino agora salva o estado completo ao final de CADA época:
     - model_state_dict
     - optimizer_state_dict
     - scheduler_state_dict
     - histórico de métricas (train_loss, train_acc, val_acc)
     - melhor val_acc até agora

   Se a energia cair no meio do epoch 7 do treino "color", ao reiniciar
   o notebook e re-executar a célula 12, ela detecta model_color_resume.pt
   e continua do epoch 8. O trabalho dos epochs 1-6 não é perdido.

2. MARCADORES "DONE" (skip-if-exists)
   Quando um treino termina completamente, o arquivo _resume.pt é apagado
   e criado um _DONE.pt com o histórico final. Da próxima vez que a célula
   rodar, ela detecta o DONE e pula automaticamente — exatamente como o
   projeto da IC faz com os arquivos .pkl.

   Arquivos de controle:
     checkpoints/model_color_DONE.pt      → treino color completo, pula
     checkpoints/model_grayscale_DONE.pt  → pula
     checkpoints/model_segmented_DONE.pt  → pula
     checkpoints/ensemble_meta.pkl        → meta-classificador já treinado, pula
     checkpoints/{train,test}_probs_{ver}.npy → probabilidades já calculadas, pula

3. CÉLULAS SEPARADAS POR MODELO
   Antes: 1 célula com ThreadPoolExecutor para os 3 modelos
   Depois: 3 células independentes (células 12, 13, 14 — uma por versão)

   Vantagem: se "color" travar na energia, "grayscale" e "segmented" podem
   ser retomados individualmente na mesma sessão.

4. LOADER "TRAIN_EVAL" SEPARADO
   O original tinha um bug sutil: ao calcular probabilidades do train set
   para o ensemble, usava o loader com augmentation (flip, rotation, jitter).
   Augmentation gera previsões diferentes a cada passagem — ruído para o
   meta-classificador.

   Agora há um 3º loader por versão (train_eval) com eval_transforms
   (sem augmentation), usado exclusivamente para calcular as features do ensemble.

5. EDA E PLOTS: SKIP-IF-EXISTS
   Todas as visualizações verificam se o PNG já existe antes de gerar.
   Poupa ~3-5 minutos em re-execuções.

6. PROBABILIDADES SEPARADAS POR VERSÃO (.npy)
   Os outputs softmax de cada modelo no train e test set são salvos em
   arquivos .npy individuais. Isso permite retreinar só o ensemble (célula 20)
   sem precisar carregar os 3 modelos grandes na memória novamente.

PROBLEMAS ENCONTRADOS E CORREÇÕES NO NOTEBOOK
----------------------------------------------

1. CÉLULA 12 (treino) TRAVAVA MESMO COM DONE FILES
   Problema: o Ray criava os 3 actors (rodava __init__ completo com DataLoaders)
   antes de detectar os DONE files dentro do train(). Com Windows + Jupyter,
   isso travava indefinidamente.
   Fix: check de DONE files ANTES de criar qualquer actor Ray. Se todos existem,
   carrega históricos direto dos DONE.pt e define trainers = None.

2. NUM_WORKERS=4 TRAVAVA NO WINDOWS + JUPYTER
   Problema: multiprocessing do DataLoader exige if __name__ == '__main__' que
   o Jupyter não tem. GPU e CPU ficavam em <10% — gargalo era I/O single-thread.
   Fix: NUM_WORKERS=0 no notebook para evitar travamento.

3. CÉLULA 15 (curvas): histories não definido após reinício de kernel
   Fix: carrega histories dos DONE.pt automaticamente se não estiver no escopo.
   Também trocado dir() por globals() para funcionar corretamente em notebooks.

4. CÉLULA 18 (probabilidades): travou por horas via Ray actor
   Problema: Ray actor do segmented bloqueou após color e grayscale terminarem.
   Fix: removido Ray da célula de probabilidades — cálculo local direto.

5. ENSEMBLE: ValueError ao concatenar arrays de tamanhos diferentes
   Problema: os 3 splits têm filenames completamente diferentes entre versões
   (UUIDs distintos), tornando impossível alinhar por imagem para stacking.
     color:     43.596 train
     grayscale: 43.203 train
     segmented: 42.984 train
   Fix: abandonado LogisticRegression (stacking). Adotado ensemble por MÉDIA
   DE SOFTMAX — todos os 3 modelos avaliam o mesmo color_test.txt, as
   probabilidades são calculadas e a média é tirada por classe.

6. AVALIAÇÃO INDIVIDUAL: test sets inconsistentes
   Problema: cada modelo era avaliado no seu próprio test set (versão específica),
   mas o ensemble usava color_test.txt — comparação injusta.
   Fix: todos os 3 modelos individuais agora avaliam no color_test.txt também.

CRIAÇÃO DO SCRIPT train.py
---------------------------
Motivação: NUM_WORKERS=0 no notebook causou gargalo severo de I/O (GPU <10%).
Para o cluster de 2 nós, é necessário script .py onde num_workers=4 funciona.

Diferenças do train.py em relação ao notebook:
  - NUM_WORKERS = 4 (funciona em script .py no Windows)
  - num_gpus=1 por actor (com 2 PCs, cada actor ocupa 1 GPU inteira)
  - Argumento --address para conectar ao cluster: python train.py --address=auto
  - Checkpoints salvos em checkpoints/ (separado do checkpoints_ipynb/ do notebook)
  - Pergunta de pesquisa atualizada: foco em tempo de treinamento distribuído
    além da comparação de acurácia ensemble vs individual

2. STACKING EXPLORATORIO — train.py
   Adicionado bloco de stacking após o softmax ensemble existente.

   MOTIVAÇÃO
   O softmax ensemble é uma comparação justa (todos os modelos veem o mesmo
   color_test.txt). Mas o stacking (LogisticRegression sobre as 114 features
   softmax) era impossível antes porque os splits de treino das 3 versões têm
   UUIDs distintos — impossível alinhar imagem a imagem.

   SOLUÇÃO EXPLORATÓRIA
   Todos os 3 modelos aceitam imagens RGB 224×224. Portanto é possível rodar
   grayscale e segmented em imagens coloridas (input "errado") para gerar
   meta-features alinhadas:

     color_train.txt → modelo color    → probs (N_train, 38) ┐
     color_train.txt → modelo grayscale → probs (N_train, 38) ┼→ X_train (N_train, 114)
     color_train.txt → modelo segmented → probs (N_train, 38) ┘

     LogisticRegression.fit(X_train, train_labels)

     color_test.txt → 3 modelos → X_test (N_test, 114)
     LogisticRegression.predict(X_test) → avalia vs test_labels_all

   NÃO É UMA COMPARAÇÃO JUSTA porque grayscale e segmented recebem imagens
   coloridas (diferente do seu treino). É declarado explicitamente no output
   e no gráfico (barra "stacking*" com nota "não justo").

   CHECKPOINT: resultado salvo em checkpoints/stacking_meta.pkl — skip se existe.

   MUDANÇAS NO train.py:
     - Adicionados imports: LogisticRegression (sklearn), joblib
     - Bloco de stacking com skip-if-exists
     - Gráfico final atualizado: 5 barras (color, grayscale, segmented,
       softmax, stacking*) com asterisco na última
     - Conclusão final imprime ganho de ambos os ensembles separadamente
       com classificação (SIM / MARGINAL / NÃO / PROMISSOR)
